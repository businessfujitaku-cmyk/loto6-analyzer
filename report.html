◆ 設計リファクタリング（全6項目）

1. データソースの一本化 修正

  • 問題：みずほ銀行公式CSV（常時403 Forbidden）とtakarakuji-official.jp HTML（Incapsula保護で0件取得）の2つのフォールバックが実働していなかった
  • 対応：死コードを完全削除し、実際に動作していた mk-mode.com/rails/loto/LOTO6_ALL.csv を唯一のデータソースとして一本化
  • 結果：scraper.py が約200行 → 約60行に簡素化。エラーの温床を排除

2. analyzer / predictor の役割分離 修正

  • 問題：奇偶比・連続性・合計値が analyzer（個別数字スコア）と predictor（組み合わせ評価）の両方で二重カウントされていた。個別数字に分解すると全奇数が同値・全偶数が同値という情報量の落ちた特徴量になっていた
  • 対応：
    • 個別数字スコア（analyzer）= freq(40%) + gap(40%) + recent(20%) の3要素のみ
    • 奇偶比・連続性・合計値は predictor 側の _fitness() 関数（組み合わせ評価）でのみ使用
  • 結果：ロジックの意味が明確に。スコアの説明可能性が向上

3. ヒートマップ Lv 配分の上位傾斜 変更

  • 問題：均等5分割（各8〜9個）では「期待度高」が多すぎて視覚的メリハリが出ない
  • 対応：上位に傾斜をかけた配分に変更

4. インメモリキャッシュ実装 新規

  • 問題：1ページロードで /api/expectation と /api/predictions が別々に analyzer を呼び、同一計算が重複実行されていた
  • 対応：latest_round をキーとしたスレッドセーフなインメモリキャッシュ（辞書でのメモ化）を実装。新規データ挿入時に自動無効化
  • 結果：初回 178ms → キャッシュヒット 4ms（約45倍高速化）

5. 空DB時のグレーアウト表示 新規

  • 問題：データ未同期時にもカラフルなヒートマップが表示される誤誘導があった
  • 対応：ready=false 時は API が items:[] / tickets:[] を返し、フロントはパルスアニメーション付きのグレースケルトンを表示
  • 結果：データ不足時にもクラッシュせず、適切な案内表示を実現

6. 表記の客観化 & 免責事項 変更

  • 画面上の「当選予測」→「出現傾向シミュレーション」に変更
  • フッターにスマートな免責事項を追加：統計的傾向の可視化であり、将来の確率を保証するものではない旨を明記
◆ 期待度スコア算出ロジック（リファクタリング後）

個別数字スコア（analyzer.py）

要素重み内容
freq40%通算出現頻度（min-max正規化）。多く出ている数字ほど高スコア
gap40%直近出現からの経過回数（正規化）。しばらく出ていない数字を拾い上げ
recent20%直近50回での出現回数。最近の勢い・トレンドを反映

score(n) = 0.40 * norm(freq[n]) + 0.40 * norm(gap[n]) + 0.20 * norm(recent[n])

組み合わせフィットネス（predictor.py）

制約評価内容
奇偶比6個中の奇偶バランスが過去最頻分布に近いか評価
連続性連続ペア（例：3-4）の発生数が過去傾向に適合するか
合計値6数字の合計が過去平均（約130前後）± 1σ に収まるか
平均期待度含まれる6数字の個別スコア平均

6000候補を重み付きサンプリング → フィットネスランキング → Jaccard類似度 ≤ 0.4 の多様化選抜で5口を確定

◆ 5口生成アルゴリズム
1

重み付きサンプリング

score(n)^1.5 を確率重みに、6数字の組み合わせを6,000通り生成（Efraimidis-Spirakis アルゴリズム）

2

統計適合スコア付与

各候補に対し、奇偶比適合 + 合計値適合 + 連続ペア適合 + 平均期待度の複合フィットネスを算出

3

多様化選抜（Greedy）

フィットネス上位から、既選抜口とのJaccard類似度が0.4未満になる5組を選定。口同士の数字被りを抑制
